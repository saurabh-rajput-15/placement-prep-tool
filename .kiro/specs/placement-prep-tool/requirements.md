# Requirements Document

## Introduction

The Placement Prep Tool is a frontend-only, offline-first single-page web application (SPA) that helps college students track and measure their placement preparation. It consolidates progress across five pillars — DSA, Projects, CS Subjects, Aptitude, and Resume — into a weighted placement readiness score. The application also provides a recommendation engine and a company simulator so students can identify gaps relative to specific target companies (Google, Amazon, TCS, Infosys). All data is persisted in the browser's `localStorage`; no backend, authentication, or cloud sync is required for the MVP.

## Glossary

- **Application**: The Placement Prep Tool single-page web application.
- **Dashboard**: The main overview screen that aggregates scores, the readiness band, and quick recommendations.
- **DSA_Tracker**: The module where users log DSA (Data Structures & Algorithms) practice topics and solved question counts.
- **Projects_Tracker**: The module where users log software projects with metadata such as difficulty, links, and status.
- **CS_Tracker**: The module where users track completion status for topics within core Computer Science subjects.
- **Aptitude_Tracker**: The module where users log aptitude practice sessions by type, difficulty, and questions solved.
- **Resume_Tracker**: The module where users fill in profile links and resume quality signals.
- **Score_Engine**: The component that computes individual module scores and the weighted Final Readiness Score.
- **Recommendation_Engine**: The component that evaluates module scores and surfaces prioritised improvement suggestions.
- **Company_Simulator**: The component that compares a user's current profile against a selected company's requirements and reports a company-specific readiness percentage and actionable gaps.
- **LocalStorage**: The browser-native `localStorage` API used for client-side data persistence.
- **Readiness_Band**: A categorical label (Beginner / Moderate / Strong / Interview Ready) derived from the Final Readiness Score.
- **Final_Readiness_Score**: The weighted aggregate score computed as `DSA×0.35 + Projects×0.25 + CS×0.20 + Aptitude×0.10 + Resume×0.10`.
- **DSA_Score**: `min(totalDSASolved / 100 × 100, 100)` — capped at 100.
- **Projects_Score**: `min(totalProjects / 5 × 100, 100)` — capped at 100.
- **CS_Score**: `completedTopics / totalTopics × 100`.
- **Aptitude_Score**: `min(totalAptitudeSolved / 200 × 100, 100)` — capped at 100.
- **Resume_Score**: `completedSections / totalSections × 100`.

## Requirements

### Requirement 1: Dashboard Overview

**User Story:** As a student, I want a single dashboard that shows my overall readiness and a summary of each preparation category, so that I can quickly understand my current placement readiness at a glance.

#### Acceptance Criteria

1. THE Dashboard SHALL display the Final_Readiness_Score as a percentage rounded to one decimal place.
2. THE Dashboard SHALL display each of the five module scores (DSA_Score, Projects_Score, CS_Score, Aptitude_Score, Resume_Score) as individual percentage values.
3. WHEN the Final_Readiness_Score is between 0 and 39 (inclusive), THE Dashboard SHALL display the Readiness_Band label "Beginner".
4. WHEN the Final_Readiness_Score is between 40 and 59 (inclusive), THE Dashboard SHALL display the Readiness_Band label "Moderate".
5. WHEN the Final_Readiness_Score is between 60 and 79 (inclusive), THE Dashboard SHALL display the Readiness_Band label "Strong".
6. WHEN the Final_Readiness_Score is between 80 and 100 (inclusive), THE Dashboard SHALL display the Readiness_Band label "Interview Ready".
7. THE Dashboard SHALL display at least one quick recommendation generated by the Recommendation_Engine whenever any module score is below its improvement threshold.
8. WHEN the user navigates to the Dashboard, THE Dashboard SHALL reflect the most recently saved data from LocalStorage without requiring a page reload.

---

### Requirement 2: DSA Tracker

**User Story:** As a student, I want to log my DSA practice by topic with solved question counts and difficulty levels, so that I can monitor my coding preparation and see it reflected in my readiness score.

#### Acceptance Criteria

1. THE DSA_Tracker SHALL allow the user to add a DSA entry with the fields: topic name (text), number of questions solved (non-negative integer), and difficulty (one of: Easy, Medium, Hard).
2. WHEN a DSA entry is added or updated, THE DSA_Tracker SHALL persist the updated entries list to LocalStorage immediately.
3. THE DSA_Tracker SHALL allow the user to delete an existing DSA entry.
4. WHEN a DSA entry is deleted, THE DSA_Tracker SHALL persist the updated entries list to LocalStorage immediately.
5. THE Score_Engine SHALL compute DSA_Score as `min(totalDSASolved / 100 × 100, 100)`, where `totalDSASolved` is the sum of solved questions across all DSA entries.
6. IF the user submits a DSA entry with an empty topic name or a negative number of solved questions, THEN THE DSA_Tracker SHALL display a validation error message and SHALL NOT save the invalid entry.

---

### Requirement 3: Projects Tracker

**User Story:** As a student, I want to log my software projects with relevant metadata, so that I can track project completion and improve my projects score.

#### Acceptance Criteria

1. THE Projects_Tracker SHALL allow the user to add a project entry with the fields: project name (text), difficulty level (one of: Basic, Intermediate, High Level), GitHub link (URL or empty), live link (URL or empty), tech stack (text), and status (one of: Planned, In Progress, Completed).
2. WHEN a project entry is added or updated, THE Projects_Tracker SHALL persist the updated entries list to LocalStorage immediately.
3. THE Projects_Tracker SHALL allow the user to edit an existing project entry.
4. THE Projects_Tracker SHALL allow the user to delete an existing project entry.
5. WHEN a project entry is deleted, THE Projects_Tracker SHALL persist the updated entries list to LocalStorage immediately.
6. THE Score_Engine SHALL compute Projects_Score as `min(totalProjects / 5 × 100, 100)`, where `totalProjects` is the count of all project entries regardless of status.
7. IF the user submits a project entry with an empty project name, THEN THE Projects_Tracker SHALL display a validation error message and SHALL NOT save the invalid entry.

---

### Requirement 4: CS Subjects Tracker

**User Story:** As a student, I want to track my study progress for individual topics within core Computer Science subjects, so that I can systematically complete my CS preparation.

#### Acceptance Criteria

1. THE CS_Tracker SHALL allow the user to add a CS topic entry with the fields: subject name (text, e.g. DBMS, OS, Computer Networks, OOP, SQL), topic name (text), and status (one of: not-started, in-progress, completed).
2. WHEN a CS topic entry status is updated, THE CS_Tracker SHALL persist the change to LocalStorage immediately.
3. THE CS_Tracker SHALL allow the user to delete an existing CS topic entry.
4. WHEN a CS topic entry is deleted, THE CS_Tracker SHALL persist the updated entries list to LocalStorage immediately.
5. THE Score_Engine SHALL compute CS_Score as `completedTopics / totalTopics × 100`, where `completedTopics` is the count of entries with status "completed" and `totalTopics` is the total count of all CS topic entries.
6. WHEN there are zero CS topic entries, THE Score_Engine SHALL treat CS_Score as 0.
7. IF the user submits a CS topic entry with an empty subject name or an empty topic name, THEN THE CS_Tracker SHALL display a validation error message and SHALL NOT save the invalid entry.

---

### Requirement 5: Aptitude Tracker

**User Story:** As a student, I want to log my aptitude practice sessions with question counts and optional accuracy scores, so that I can track my aptitude readiness.

#### Acceptance Criteria

1. THE Aptitude_Tracker SHALL allow the user to add an aptitude entry with the fields: aptitude type (one of: Quantitative, Logical, Verbal), difficulty level (one of: Easy, Medium, Hard), questions solved (non-negative integer), and accuracy or mock score (optional, a number between 0 and 100 inclusive).
2. WHEN an aptitude entry is added or updated, THE Aptitude_Tracker SHALL persist the updated entries list to LocalStorage immediately.
3. THE Aptitude_Tracker SHALL allow the user to delete an existing aptitude entry.
4. WHEN an aptitude entry is deleted, THE Aptitude_Tracker SHALL persist the updated entries list to LocalStorage immediately.
5. THE Score_Engine SHALL compute Aptitude_Score as `min(totalAptitudeSolved / 200 × 100, 100)`, where `totalAptitudeSolved` is the sum of questions solved across all aptitude entries.
6. IF the user submits an aptitude entry with a negative number of solved questions or an accuracy value outside the range 0–100, THEN THE Aptitude_Tracker SHALL display a validation error message and SHALL NOT save the invalid entry.

---

### Requirement 6: Resume Tracker

**User Story:** As a student, I want to fill in my profile links and resume quality signals, so that the tool can calculate my resume score and suggest what is missing.

#### Acceptance Criteria

1. THE Resume_Tracker SHALL provide input fields for: LinkedIn URL, GitHub URL, Portfolio URL, highest project difficulty level (one of: Basic, Intermediate, High Level, or empty), achievements (text), and activities or positions of responsibility (text).
2. WHEN the user saves the resume form, THE Resume_Tracker SHALL persist all resume fields to LocalStorage immediately.
3. THE Score_Engine SHALL compute Resume_Score as `completedSections / totalSections × 100`, where `totalSections` is 6 (LinkedIn, GitHub, Portfolio, project level, achievements, activities) and `completedSections` is the count of those fields that are non-empty.
4. THE Resume_Tracker SHALL display the current Resume_Score to the user.
5. IF the user provides a LinkedIn, GitHub, or Portfolio value that does not begin with "http://" or "https://", THEN THE Resume_Tracker SHALL display a validation warning and SHALL still allow the user to save the value.

---

### Requirement 7: Score Engine — Weighted Readiness Calculation

**User Story:** As a student, I want the application to calculate my overall placement readiness score using a transparent weighted formula, so that I trust the score and understand how each category contributes.

#### Acceptance Criteria

1. THE Score_Engine SHALL compute Final_Readiness_Score as `DSA_Score × 0.35 + Projects_Score × 0.25 + CS_Score × 0.20 + Aptitude_Score × 0.10 + Resume_Score × 0.10`.
2. THE Score_Engine SHALL cap each module score at 100 before applying weights.
3. THE Score_Engine SHALL recompute all scores whenever any tracker data changes.
4. THE Score_Engine SHALL produce a Final_Readiness_Score in the range 0 to 100 (inclusive).
5. THE Score_Engine SHALL expose the individual module scores and the Final_Readiness_Score to the Dashboard, Recommendation_Engine, and Company_Simulator.

---

### Requirement 8: Recommendation Engine

**User Story:** As a student, I want to receive actionable improvement suggestions based on my current scores, so that I always know what to work on next.

#### Acceptance Criteria

1. WHEN DSA_Score is less than 60, THE Recommendation_Engine SHALL generate the recommendation: "Solve more topic-based DSA questions to strengthen your coding skills."
2. WHEN Projects_Score is less than 60, THE Recommendation_Engine SHALL generate the recommendation: "Build or finish more projects to improve your projects score."
3. WHEN CS_Score is less than 70, THE Recommendation_Engine SHALL generate the recommendation: "Complete incomplete core CS subject topics (DBMS, OS, Networks, OOP)."
4. WHEN Aptitude_Score is less than 50, THE Recommendation_Engine SHALL generate the recommendation: "Practise more aptitude questions across Quantitative, Logical, and Verbal sections."
5. WHEN Resume_Score is less than 100, THE Recommendation_Engine SHALL generate the recommendation: "Fill in missing resume sections: links, achievements, and activities."
6. WHEN all module scores meet or exceed their respective thresholds, THE Recommendation_Engine SHALL display the message: "Great job! Your profile is well-rounded. Focus on mock interviews."
7. THE Recommendation_Engine SHALL display all applicable recommendations simultaneously when multiple thresholds are not met.

---

### Requirement 9: Company Simulator

**User Story:** As a student, I want to compare my current preparation profile against the requirements of a specific target company, so that I can understand exactly what gaps I need to close.

#### Acceptance Criteria

1. THE Company_Simulator SHALL support the following companies: Google, Amazon, TCS, and Infosys.
2. WHEN the user selects a company, THE Company_Simulator SHALL display that company's expected thresholds (e.g., minimum DSA questions, minimum projects, minimum CGPA, required CS subjects).
3. THE Company_Simulator SHALL compute a company-specific readiness percentage by comparing the user's current metrics against the selected company's thresholds.
4. THE Company_Simulator SHALL display a list of gaps — specific areas where the user's current metrics fall below the selected company's thresholds.
5. THE Company_Simulator SHALL display at least one actionable suggestion for each identified gap.
6. THE Company_Simulator SHALL allow the user to input their CGPA (a decimal number between 0.0 and 10.0 inclusive) and SHALL include CGPA in the gap analysis.
7. WHEN the user selects a different company, THE Company_Simulator SHALL update the readiness percentage, gap list, and suggestions without requiring a page reload.
8. IF the user enters a CGPA value outside the range 0.0–10.0, THEN THE Company_Simulator SHALL display a validation error and SHALL NOT use the invalid value in calculations.

---

### Requirement 10: Data Persistence

**User Story:** As a student, I want my preparation data to be saved automatically in my browser, so that I do not lose progress when I close or refresh the page.

#### Acceptance Criteria

1. THE Application SHALL store all tracker data (DSA entries, project entries, CS topic entries, aptitude entries, resume fields, CGPA, and selected company) in LocalStorage under a single namespace key.
2. WHEN the Application loads, THE Application SHALL read all data from LocalStorage and restore the UI state to reflect the previously saved data.
3. WHEN any tracker data is created, updated, or deleted, THE Application SHALL write the complete updated data object to LocalStorage within 500 milliseconds of the change.
4. WHEN LocalStorage is unavailable (e.g., private browsing with storage blocked), THE Application SHALL display a warning message informing the user that data will not be persisted, and SHALL continue to operate in-memory for the current session.
5. THE Application SHALL NOT transmit any user data to external servers or third-party services.

---

### Requirement 11: Single-Page Navigation

**User Story:** As a student, I want to switch between the Dashboard, each tracker, and the Company Simulator using a navigation menu, so that I can access any section without a full page reload.

#### Acceptance Criteria

1. THE Application SHALL provide a navigation menu with links to: Dashboard, DSA Tracker, Projects Tracker, CS Subjects Tracker, Aptitude Tracker, Resume Tracker, and Company Simulator.
2. WHEN the user clicks a navigation link, THE Application SHALL display the corresponding section and hide all other sections without reloading the page.
3. THE Application SHALL highlight the currently active navigation link to indicate the user's current location.
4. WHEN the Application first loads with no stored state, THE Application SHALL display the Dashboard section by default.
